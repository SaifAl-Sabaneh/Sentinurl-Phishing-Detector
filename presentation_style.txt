SentinURL: Project Architecture & Defense Masterclass
The Complete Step-by-Step Chronological Guide to How We Built This Project

Phase 1: The Core Concept & Data Engineering
The Problem We Solved: Traditional firewalls like Google Safe Browsing (GSB) use reactive 'blacklists'. If a new phishing site was created 5 minutes ago, GSB won't block it because it hasn't seen it yet. We wanted to build an 'Artificial Intelligence' that could mathematically predict if a site was malicious, even if it had never been seen before (Zero-Day Attacks).
Getting the Data: We didn't use a small, toy dataset. We sourced a massive, raw dataset containing over 1,000,000 recorded URLs. We had to aggressively clean this data: fixing broken text encodings (like utf-8/latin1), dropping empty rows, and separating the explicitly 'Phishing' links from the 'Safe' links so the AI could learn the difference.
Phase 2: Training Stage 1 (The NLP Brain)
Because scammers use specific psychological words ('login', 'secure', 'account-update'), we needed an engine that could read the URL like a sentence.
The Technology: Term Frequency-Inverse Document Frequency (TF-IDF) Vectorizer mapped to a Logistic Regression Model.
How we built it: We programmed the TF-IDF vectorizer to extract 'n-grams' (chunks of text up to 7 characters long) from the URLs. This allowed the engine to memorize up to 150,000 of the most dangerous phonetic syllables and text structures. When a URL passes through Stage 1, this NLP model instantly calculates the linguistic probability that the URL is trying to deceive someone.

Phase 3: Training Stage 2 & Feature Engineering
Sometimes, scammers use completely random, innocent-looking text. To catch these, we couldn't just read the words; we had to measure the structural DNA of the link itself.
The Technology: HistGradientBoosting Classifier. A highly robust, fast decision-tree algorithm.
Feature Engineering (enhanced_original.py): We wrote custom Python logic to extract deep mathematical metrics from every URL:
  • Character Entropy & Vowel Ratios: Botnets register random domains like 'q709glbpxn0zeknc'. We programmed the engine to calculate the lack of vowels and flag it as high-entropy gibberish.
  • Subdomain Nesting: Phishers hide the real threat deep in subdomains (e.g., 'security-update.paypal.com.scam-host.xyz'). We count the dots and slashes to catch this.
  • Categorized Keywords: We physically split 'suspicious words' into specific arrays: Financial ('crypto', 'wallet'), Scam ('hack', 'free'), and Credential ('auth', 'login').
  • Brand Impersonation: We check if Fortune 500 brands (Google, Netflix, Apple) are illegitimately hiding in the URL path.
Phase 4: The Ultimate Pipeline (sentinurl.py)
With two separate AI models trained, we needed a 'Master Controller' to weave them together and apply real-world logic. This became 'sentinurl.py'.
The Layered Defense Architecture:
  1. Quick Allowlists: If the domain is officially Jordanian government or a top Alexa 500 site, clear it instantly to save processing power.
  2. Live API Verification: Query Google Safe Browsing and check the live TLS Certificate and Domain Age (WHOIS).
  3. The ML Fusion: The engine checks Stage 1 (Text) and Stage 2 (Structure) and merges their scores based on certainty levels.
  4. Brand Overrides: If the AI score is low, but the URL explicitly mimics 'PayPal', the Brand Deception rule forcefully overwrites it to 'PHISHING'.
How we fixed the massive False Negative Loophole (March 11th Patch): During bulk testing, we realized the AI was missing 11% of attacks. We traced the bug to rigid 'Fail-Safe Overrides'. The legacy system had a rule: 'If Google Safe Browsing hasn't flagged it yet, just trust it and set the score to 0.0'. We completely dismantled this flaw! We wrote patches enforcing that the ML AI's threat confidence (up to 75%) must strictly override Google's blindness. We also patched a vulnerability where hacked university '.edu' and '.gov' domains were being automatically cleared as safe. This single session of debugging skyrocketed our True Positive Accuracy to 98.4%!

Phase 5: The Streamlit Web Dashboard
An AI is useless if the user cannot interact with it. We built 'streamlit_app.py' as a seamless, high-performance web frontend.
Why Streamlit?: It allowed us to rapidly build interactive, data-driven dashboards natively in Python without managing complex React or HTML backends.
What we developed:
  • Single Scanner: A tab detailing the real-time breakdown of a URL, displaying interactive gauges for Stage 1 vs Stage 2 confidence, and rendering an HTML footprint report.
  • Bulk Scanner: We built a multi-threaded data ingestion engine. Users can upload a CSV of 1,000 URLs, and the system processes them simultaneously.
  • The Analytics Metric Dashboard: We literally injected a real-time mathematical panel directly into the UI. The second a bulk scan finishes, it renders four dynamic statistic boxes (Total Scanned, Phishing Detected, Safe URLs) that calculate the exact effectiveness percentage instantly.
